Gap analysis — systems not addressed in the solution design
Blocker PLM is marked "Product Master" — but Conscia is also being positioned as product master
The architecture diagram explicitly labels PLM (TBD) as the Product Master. The Conscia solution design's stated strategic objective is to make DX Graph the product master for configurable products. These two are in direct conflict. If Lovesac procures a PLM, it will either replace DX Graph's role or create a duplicate source of truth. This needs to be resolved at kickoff: is Conscia the product master instead of a future PLM, or is DX Graph a compatibility/orchestration layer that sits below a PLM? The answer changes what DX Graph owns and what it receives from upstream.
Blocker Celigo (Integration Hub) sits between source systems and everything else — including Conscia
The architecture shows Celigo as the Integration Hub through which all system-to-system data flows. The solution design describes DX Graph Jobs pulling data from "up to three source backends" but doesn't mention Celigo once. This is a critical gap: will DX Graph ingest directly from SAP/Magento/etc., or via Celigo? If via Celigo, the data format, transformation logic, and delivery cadence are Celigo's concern — not Conscia's. The Magento export CSV in the project folder already has Celigo in the filename (CeligoMagento2ItemExportSearch...), confirming Celigo is already in the data pipeline today.
Blocker OMS (Manhattan Active Omni) is the Order Master — BOM explosion output lands here
The solution design's BOM Explosion flow produces OMS-shaped output, but Manhattan Active Omni is not mentioned by name. The BOM Explosion flow sends "fulfilment-optimised BOM" to OMS — but how does it get there? Does DX Engine call Manhattan directly? Does it go via Celigo? Does Lovesac's OMS team need to build a receiver? This integration path is undefined and it's a delivery-critical one — the whole POS use case depends on the Config ID → BOM → OMS chain working.
Risk Threekit (Visual Configurator) is the current rules source — not just a UI tool
Threekit appears in the architecture as a supporting commerce system. The solution design treats it only as the source of hard-coded compatibility rules to be replaced. But Threekit is also Lovesac's live customer-facing configurator (as documented in the Builder Guide). In Phase 1, DX Graph and DX Engine will coexist with Threekit — Threekit won't be decommissioned. There needs to be an explicit plan for the transition period: does Threekit continue to serve the web configurator while DX Engine is being built? When does the switchover happen? Who owns the Threekit rules until then?
Risk PIM is listed as TBD — Conscia may be filling this role without it being formally scoped that way
The architecture shows a PIM box as TBD in the commerce supporting systems layer. From context, Conscia DX Graph is effectively being used as the PIM for configurable products. This is fine, but if Lovesac later procures a traditional PIM (e.g. Akeneo, Salsify), it will need to integrate with or replace DX Graph. This future dependency should be noted in the solution design so the data model is built to be PIM-interoperable, not PIM-replacing in ways that would be painful to unwind.
Risk SF Retail Cloud (POS) is the named POS system — solution design calls it generic "POS Channel"
The solution design refers to "POS Channel" and "POS terminals" throughout, but the architecture names the system as Salesforce Retail Cloud. The Conscia BFF research doc includes Salesforce Commerce Cloud connectors. The team should confirm: does Conscia have a documented Salesforce Retail Cloud integration pattern? The API shape that SF Retail Cloud expects may differ from a generic POS. This affects how the BOM Explosion and Config Validation Orchestration Flows are shaped.
Risk AgentForce appears twice — as AI Agent channel AND as the AI Data Layer
Salesforce AgentForce is shown at the top of the diagram as a customer-facing AI Agent channel, and again at the bottom as the AI Data Layer. The solution design mentions Google Shopping and ChatGPT as agentic channels but doesn't mention AgentForce. If AgentForce is Lovesac's strategic AI agent platform, it should be the primary agentic channel in the solution design — not an afterthought. The MCP tooling that DX Engine exposes would need to be compatible with AgentForce's agent invocation model. Clarify with Lovesac: is AgentForce the primary AI agent or one of several?
Risk Inventory management (NetSuite) is out of scope but affects compatibility logic
Inventory Mgmt. is shown as TBD/NetSuite in the enterprise layer. The solution design explicitly excludes real-time inventory from Phase 1 scope. However, Quickship availability is an inventory concept — if Quickship reflects live stock in NetSuite rather than a product classification, the Quickship Experience Rule will need a real-time inventory call that isn't currently designed. This connects directly to Open Question 6 in the solution design.
Note ERP is NetSuite — not SAP as assumed in the solution design
The solution design mentions "an SAP backend" when describing what Conscia won't connect to directly (Section 3.3). But Lovesac's ERP is NetSuite, not SAP. The BOM output from DX Engine is described as going to SAP, but it should be NetSuite. This is a terminology error in the solution design that should be corrected before the kickoff deck is presented.
Note Mobile Apps (StealthTech v2.0) is a channel not mentioned in the solution design
A Mobile App channel (specifically called out as StealthTech v2.0) appears at the top of the architecture. This is presumably the companion app for StealthTech audio/charging system management. It is not a configurable commerce channel, so it's not a gap in the Conscia scope — but it's worth confirming: does the StealthTech app need to query product compatibility data? If StealthTech components need to be validated as compatible with a Sactional configuration, DX Engine would be the right layer for that too.
